# FlashDeal — 한정 수량 플래시 세일 플랫폼

> 대규모 동시 요청 환경에서의 동시성 제어 및 비동기 처리 아키텍처 탐구

- **스택** Java 17 · Spring Boot 3.x · MySQL · Redis · Redisson · Kafka
- **형태** 개인 토이 프로젝트

---

## 프로젝트 동기

팬덤 플랫폼, 커머스 등 실제 서비스에서 "특정 시간에 수천 명이 동시에 몰리는" 트래픽 스파이크는 흔한 문제다.
관련 기업들의 기술 스택을 살펴보면서 Redis와 Kafka가 공통적으로 등장하는 것을 발견했고,
단순히 사용법을 익히는 것을 넘어 **어떤 문제 상황에서 이 기술이 필요해지는지**를 직접 재현하고 해결해보고 싶었다.

각 기술을 도입한 이유, 대안과의 비교, 실제로 측정한 before/after 수치를 기록하는 데 초점을 맞춘다.

---

## 해결하고자 한 기술적 문제

| # | 문제 | 설명 |
|---|------|------|
| P1 | **재고 오버셀 (Overselling)** | DB 트랜잭션만으로는 수백 개의 동시 요청이 몰릴 때 재고가 음수가 되는 현상을 막기 어렵다. |
| P2 | **복합 조건의 원자성** | "중복 확인 → 수량 확인 → 발급"처럼 여러 단계를 하나의 임계 구역으로 묶어야 하는 경우, 단순 원자 연산만으로는 부족하다. |
| P3 | **API 응답 지연** | 주문 성공 후 알림·정산 등 후처리를 동기로 처리하면 응답 시간이 길어지고 장애 전파 위험이 생긴다. |
| P4 | **DB 직접 타격** | 상품 조회처럼 읽기가 잦은 요청이 모두 DB로 향하면 트래픽 스파이크 시 DB 과부하가 발생한다. |

---

## 기술 선택 근거

### Redis — 두 가지 역할

**① DECR 원자 연산 (P1, P4 해결)**

단일 명령어로 읽기·차감·반환이 원자적으로 처리되어 오버셀 방지.
상품 조회 캐싱으로 DB 부하도 감소.

**② Redisson 분산 락 (P2 해결)**

코드 블록 전체를 임계 구역으로 만들어 복합 조건 로직을 직렬화.
DECR로 처리할 수 없는 다단계 비즈니스 로직에 적용.

| 구분 | Redis DECR | 분산 락 |
|------|-----------|---------|
| 적합한 상황 | 단순 숫자 증감 | 복수 조건 검증 + 처리 |
| 원자성 범위 | 단일 연산 | 임의의 코드 블록 |
| 복잡도 | 낮음 | 높음 (타임아웃, 데드락 고려) |
| 적용 예시 | 재고 차감 | 중복 구매 방지, 쿠폰 발급 |

### Kafka — 비동기 이벤트 버스 (P3 해결)

주문/발급 완료 후 이벤트를 발행하고, 알림·정산·히스토리 Consumer가 독립적으로 소비.
API 응답 시간 단축 + 서비스 간 결합 제거.
Consumer 실패 시 Dead Letter Topic으로 격리해 장애 내성 확보.

## 면접 스토리 흐름

```
① DB 트랜잭션만으로 구현
        ↓ 문제: 동시 요청 시 재고 오버셀 발생 (JMeter로 증명)

② Redis DECR 원자 연산 적용
        ↓ 해결: 재고 차감 오버셀 제거
        ↓ 한계: 복수 조건(중복 확인 + 수량 확인 + 발급)을 묶을 수 없음

③ Redisson 분산 락 적용 (쿠폰 발급)
        ↓ 해결: 복잡한 비즈니스 로직 블록 전체를 직렬화
        ↓ 고민: 락 타임아웃, 데드락 방지 처리

④ Kafka로 후처리 비동기화
        ↓ 해결: API 응답 속도 개선, 서비스 간 결합도 제거
        ↓ 추가: Dead Letter Topic으로 장애 내성 확보
```
